# DocMaster PM - 飞书文档整理大师

## 适用范围

当用户提供飞书、Lark、豆包文档链接或文档 token，并表达以下任一意图时使用本 skill：

- 整理文档、优化排版、结构化内容、重写成专业文档。
- 把会议记录、访谈记录、碎片笔记整理成纪要、需求、方案或行动计划。
- 从混乱内容中提炼背景、目标、问题、方案、风险、待办。
- 以产品经理、项目经理、运营负责人或业务负责人视角重塑文档。
- 要求直接更新原飞书文档，或创建一份整理后的新文档。

不适用：

- 只需要读取或摘要飞书文档时，优先使用 `lark-doc`。
- 需要批量管理云空间文件、权限、下载上传时，优先使用 `lark-drive`。
- 需要在多维表格、电子表格或幻灯片中处理内容时，切换到对应 lark skill。

## 前置检查

1. 先阅读并遵循 `../lark-shared/SKILL.md` 的认证和错误处理要求。
2. 文档读写通常需要：
   - 读取：`docx:document:readonly`
   - 写入：`docx:document`
   - 登录：`lark-cli auth login --domain docx`
3. 如果用户只说“帮我整理这个文档”，默认先读取文档并产出整理方案；只有用户明确要求“写回、覆盖、更新原文档、生成新文档”时才执行写操作。

## 核心原则

1. **信息零丢失**：不得删除、省略或过度概括原文中的实质信息。看似零散的内容也要归入合适模块。
2. **只增不减**：可以补充标题、过渡句、小结、背景推导、风险说明和行动项，但不能改变原始意图或编造事实。
3. **结构先行**：先判断文档类型，再选择骨架；不要把所有文档都套成同一个模板。
4. **可执行表达**：整理后的内容应让读者能快速判断背景、目标、当前状态、关键决策、待办和风险。
5. **保守写回**：写回前必须完成自检；写回失败时保留原文档，不做破坏性尝试。

## 工作流

### 1. 识别输入

从用户消息中提取：

- 文档 URL 或 token。
- 用户希望的目标：整理、排版、重构、总结、PRD、会议纪要、项目计划等。
- 是否要求写回原文档，还是只在对话中输出。
- 是否有指定风格，例如“高级产品经理”“老板可读”“研发可执行”“客户汇报”。

如果缺少文档链接或 token，直接向用户索要，不要猜测。

### 2. 读取文档

默认用 Markdown 读取，便于保留标题、列表和表格结构：

```bash
lark-cli docs +fetch --api-version v2 --doc "<文档URL或token>" --doc-format markdown
```

读取后先判断：

- 文档是否为空或只有标题。
- 原文是否包含表格、图片、附件、代码块、嵌入 Base/Sheet/Whiteboard。
- 内容主要是会议记录、需求草稿、方案说明、项目计划、复盘、调研笔记还是混合素材。

如果文档中有无法通过 Markdown 完整保留的对象，先说明风险，并优先保留文本结构；需要处理图片、附件或画板时切换到对应 skill。

### 3. 建立信息清单

在重写前先做内部梳理，把原文要点映射到下列类别：

| 类别 | 判断标准 |
| --- | --- |
| 背景 | 为什么有这件事、上下文、现状、业务问题 |
| 目标 | 希望达成的结果、衡量标准、成功定义 |
| 用户/角色 | 涉及的人群、负责人、参与方、审批方 |
| 需求/方案 | 功能、流程、规则、策略、设计想法 |
| 数据/约束 | 指标、时间、范围、技术限制、资源限制 |
| 决策 | 已确认的结论、方案取舍、会议共识 |
| 待办 | 谁在什么时间前做什么，产出是什么 |
| 风险/问题 | 未确认、阻塞、依赖、争议、潜在风险 |
| 备注素材 | 暂时无法归类但不能丢失的信息 |

任何无法判断归属的信息，放到“补充信息/原始备注”中，不要删除。

### 4. 选择文档骨架

根据内容类型选择最合适的结构。

#### 通用产品文档

```markdown
# <清晰标题>

## 一、文档摘要
## 二、背景与现状
## 三、目标与范围
## 四、核心内容/需求拆解
## 五、方案细节
## 六、风险与未决问题
## 七、行动计划
## 八、补充信息
```

#### 会议纪要

```markdown
# <会议主题>会议纪要

## 一、会议结论
## 二、背景与讨论脉络
## 三、关键议题
## 四、已确认决策
## 五、待办事项
## 六、风险与未决问题
## 七、原始备注归档
```

待办事项优先整理成表格：

| 事项 | 负责人 | 截止时间 | 交付物 | 状态 |
| --- | --- | --- | --- | --- |

#### PRD/需求说明

```markdown
# <功能/项目名>需求说明

## 一、背景
## 二、目标
## 三、用户与场景
## 四、需求范围
## 五、功能规则
## 六、流程说明
## 七、数据与埋点
## 八、边界情况
## 九、风险与依赖
## 十、验收标准
```

#### 项目计划

```markdown
# <项目名>项目计划

## 一、项目概览
## 二、目标与成功标准
## 三、里程碑
## 四、任务拆解
## 五、资源与分工
## 六、风险与依赖
## 七、同步机制
```

### 5. 重塑内容

重写时遵守：

- 标题层级不超过 4 级。
- 大段文本改成短段落、列表或表格。
- 对人名、时间、指标、系统名、结论、风险使用加粗。
- 对争议、阻塞和注意事项使用引用块。
- 不确定的信息标注为“待确认”，不要自行填补。
- 口语和重复表达可以合并，但合并后必须覆盖原始含义。
- 原文中的具体数值、时间、名称、状态不能被模糊化。

### 6. 自检

写回或交付前必须检查：

- 原文中的每个实质要点是否都出现在新文档中。
- 是否新增了没有依据的事实、结论、承诺或时间。
- 待办是否具备“事项、负责人、时间、产出”中的至少两个维度；缺失项标为“待确认”。
- 风险和未决问题是否被显式列出。
- 标题、表格、列表是否能在飞书 Markdown 中正常渲染。

### 7. 输出策略

如果用户没有要求写回：

- 在对话中输出整理后的核心版本。
- 如果内容很长，先输出结构目录和关键部分，并询问是否继续生成完整文档。

如果用户要求写回：

1. 先完成整理稿。
2. 明确说明将更新原文档或创建新文档。
3. 优先选择稳定路径写入。
4. 写入后重新读取并抽查标题、关键结论和待办是否存在。
5. 向用户返回文档链接和简短变更说明。

## 写回方式

### 路径 A：创建新文档（推荐）

适合内容较长、包含中文标点、表格、复杂 Markdown，或用户允许生成新文档时使用。

```bash
cmd /c "type .docmaster-temp.md | lark-cli docs +create --api-version v2 --title <英文或简洁标题> --content - --doc-format markdown"
```

创建后验证：

```bash
lark-cli docs +fetch --api-version v2 --doc <新文档id或token> --doc-format markdown
```

如果原文档在 Wiki 中，并且用户要求放回原路径，需要先查询原节点，再用 wiki move 移动新文档。

### 路径 B：覆盖原文档（谨慎）

仅在用户明确要求覆盖原文档，且内容规模和格式适合时使用：

```bash
lark-cli docs +update --api-version v2 --doc "<文档URL或token>" --command overwrite --doc-format markdown --content "<整理后的完整内容>"
```

限制：

- `--content` 不支持 `@file` 或 stdin `-`。
- 内容包含复杂引号、特殊字符或过长时容易失败。
- 若返回 `degrade_code=1011` 或 No document changes，改走路径 A。
- 不要反复覆盖尝试，避免误清空或破坏原文。

## 异常处理

| 情况 | 处理方式 |
| --- | --- |
| 权限不足 | 提示用户执行 `lark-cli auth login --domain docx` 或补充权限 |
| 文档读取为空 | 说明读取结果为空，询问是否仍要创建整理模板 |
| 文档过长 | 先分段整理，再汇总，不要一次性丢失中间内容 |
| overwrite 失败 | 停止覆盖，改为创建新文档 |
| 原文包含大量图片/附件 | 保留文本，提示需要 lark-drive/lark-doc 进一步处理附件 |
| 原文包含 Base/Sheet/Whiteboard | 读取文档 token 后切换到对应 skill 下钻 |

## 回复格式

完成后用简短说明收尾：

- 已整理成什么结构。
- 是否写回或新建文档。
- 是否存在未确认信息、权限问题或写回限制。
- 如果有链接，返回可点击文档链接。

避免输出“我作为高级产品经理”这类自我描述；直接交付结果。
